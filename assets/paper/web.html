UNIT 1

1. Cloud Computing with examples

Cloud computing delivers computing resources—servers, storage, databases, networking, software—over the internet on a pay-as-you-go basis.

Core attributes: on-demand availability, scalability, elasticity, reduced capital cost, global access, managed infrastructure.

Types of resources provided:
• Computing power: virtual machines, containers.
• Storage: object, file, block storage.
• Databases: SQL, NoSQL, in-memory.
• Networking: load balancers, VPN, VPC.

Examples:
• Gmail uses cloud servers to store emails and allow access anywhere.
• Netflix uses cloud infrastructure to stream content globally.
• Google Drive stores and synchronizes user files across devices.
• By using AWS EC2, a company deploys a website without owning servers.

Mechanism: cloud service providers operate massive datacenters. Users access their allocated resources via web consoles or APIs. Providers automate provisioning, scaling, monitoring, and security.

2. IaaS vs PaaS vs SaaS with real-world applications

IaaS (Infrastructure as a Service):
• Delivers raw compute, storage, and networking resources.
• User controls OS, applications, runtime.
• Use cases: hosting websites, running custom applications, backup and recovery systems.
• Examples: AWS EC2, Google Compute Engine, Azure VMs, DigitalOcean Droplets.

PaaS (Platform as a Service):
• Delivers a ready-to-use platform including OS, runtime, database, and tools.
• Developer focuses only on code and deployment, not the underlying infrastructure.
• Use cases: application development, CI/CD automation, microservices.
• Examples: AWS Elastic Beanstalk, Google App Engine, Heroku.

SaaS (Software as a Service):
• Fully managed application delivered over the internet.
• User only consumes the service; no installation or maintenance.
• Use cases: email, CRM, collaboration tools, ERP.
• Examples: Gmail, Salesforce, Microsoft 365, Zoom.

Differences summarized:
• IaaS → maximum control.
• PaaS → balanced control and convenience.
• SaaS → minimum control, maximum convenience.

3. Cloud Deployment Models

Public Cloud:
• Services delivered over the public internet by third-party providers.
• Resources are shared among multiple users.
• High scalability, low cost, easy access.
• Example: AWS, Azure, GCP.

Private Cloud:
• Dedicated infrastructure for a single organization.
• Hosted on-premises or in a private datacenter.
• High security, customization, compliance.
• Example: VMware private cloud, OpenStack deployment inside a company.

Hybrid Cloud:
• Mix of public and private cloud.
• Data and applications move between environments.
• Useful for sensitive + non-sensitive workloads.
• Example: bank keeps customer data on private cloud but uses AWS for analytics.

Community Cloud:
• Shared infrastructure for organizations with common requirements (security, compliance, mission).
• Managed by one organization or a third party.
• Example: healthcare organizations sharing a HIPAA-compliant cloud.

4. Why AWS? Its compute, storage, database, and networking options

Reasons for AWS dominance:
• Global infrastructure: multiple regions and availability zones.
• Scalability and elasticity: Auto Scaling, Load Balancers.
• Reliability: fault-tolerant architecture.
• Security: IAM, encryption, compliance.
• Pay-as-you-go pricing.
• Rich ecosystem of services ("largest cloud service catalog").

Compute services:
• EC2: virtual servers with many instance types.
• Lambda: serverless compute triggered by events.
• ECS/EKS: container orchestration for Docker/Kubernetes.
• AWS Batch: batch processing jobs.

Storage services:
• S3: scalable object storage for files, media, backups.
• EBS: block storage for EC2 instances.
• EFS: elastic file system for shared access.
• Glacier: archival cold storage.

Database services:
• RDS: managed relational databases (MySQL, PostgreSQL, Oracle, SQL Server).
• DynamoDB: serverless NoSQL key-value database.
• Aurora: high-performance relational database.
• ElastiCache: in-memory caching (Redis, Memcached).
• Redshift: data warehousing.

Networking services:
• VPC: isolated virtual network environment.
• Route 53: DNS and domain management.
• Elastic Load Balancer: distributes traffic.
• Direct Connect: private connection from on-premises to AWS.
• CloudFront: content delivery network.

5. AWS Pricing Models: On-Demand, Reserved, Spot

On-Demand Pricing:
• Pay by the hour/second for compute and storage.
• No upfront payments or long-term commitments.
• Used when workloads are unpredictable, short-term, or for testing.

Reserved Instances (RIs):
• Commit to a 1-year or 3-year term.
• Receive major discounts (up to 70% compared to On-Demand).
• Best for steady-state workloads, predictable traffic.

Spot Instances:
• Purchase unused AWS capacity at deep discounts (up to 90%).
• AWS can reclaim the instance anytime if demand increases.
• Suitable for fault-tolerant and flexible workloads (batch jobs, data processing, ML training).

Pricing logic:
• On-Demand → flexibility.
• Reserved → cost optimization for stable workloads.
• Spot → lowest cost for interruptible tasks.

UNIT 2

1. EC2 and Different Instance Types (purpose + examples)

Amazon EC2 (Elastic Compute Cloud) provides virtual servers—called instances—to run applications. EC2 gives full control over OS, storage, network, and software. Instances scale up or down based on workload.

Instance families and their purposes:

General Purpose Instances (T-series, M-series):
• Balanced CPU, memory, and network.
• Used for web servers, small databases, development environments.
• Examples: t2.micro for simple websites, m5.large for enterprise apps.

Compute-Optimized Instances (C-series):
• High-performance processors.
• Used for heavy computation: scientific modeling, ML inference, game servers.
• Example: c5.xlarge.

Memory-Optimized Instances (R-series, X-series):
• Large RAM compared to CPU.
• Used for in-memory databases, caching, real-time analytics.
• Example: r5.large, x1e.32xlarge for SAP HANA.

Storage-Optimized Instances (I-series, D-series):
• High IOPS and local NVMe SSDs.
• Used for NoSQL databases, OLTP systems, data warehousing.
• Example: i3.large.

GPU Instances (P-series, G-series):
• GPUs for parallel processing.
• Used for deep learning, 3D rendering, video processing.
• Example: p3.2xlarge, g4dn.xlarge.

Burstable Instances (T-series):
• Provide baseline performance with CPU burst capability.
• Suitable for apps with variable load.
• Example: t3.micro for WordPress hosting.

2. Steps to Launch an EC2 Instance (including AMI selection and configuration)

Step 1: Open EC2 Console
Navigate to AWS Management Console → EC2 → Launch Instance.

Step 2: Choose AMI
Select an Amazon Machine Image that defines OS and preinstalled software.
Examples: Amazon Linux AMI, Ubuntu, Windows Server, custom AMIs.

Step 3: Choose Instance Type
Select based on CPU, RAM, storage, network features.

Step 4: Configure Instance
Define:
• Number of instances
• Network (VPC, subnet)
• IAM role
• Shutdown behavior
• Monitoring (CloudWatch)
• User data script (for automatic setup)

Step 5: Add Storage
Choose EBS volume type: gp2/gp3 (general purpose), io1/io2 (high IOPS), st1/sc1 (throughput optimized).
Set volume size.

Step 6: Add Tags
Key–value identifiers for management.
Example: Name = WebServer1.

Step 7: Configure Security Group
Define inbound rules: SSH (22), HTTP (80), HTTPS (443).
Outbound traffic is usually open by default.

Step 8: Create/Select Key Pair
Used to connect securely to the instance.
Download PEM/PPK file.

Step 9: Launch Instance
Instance gets a public IP if launched in a public subnet.
Wait until state becomes running.

3. AMI and How to Create AMI from a Running Instance

AMI (Amazon Machine Image) is a template that contains:
• Operating System
• Application software
• Configuration (packages, settings)
• EBS snapshot(s)

AMI enables launching identical instances quickly.

Types:
• Public AMI
• Private AMI
• Marketplace AMI
• Customized AMI created by users

Creating AMI from a running instance:

  1. Select the instance in EC2 Console.
  2. Choose Actions → Image and Templates → Create Image.
  3. Provide name, description, choose volumes to include.
  4. AWS takes a snapshot of the root EBS volume.
  5. AMI becomes available under Images → AMIs.

Use case:
• Creating consistent environments for scaling.
• Backup before application upgrades.
• Launching multiple identical servers.

4. Auto Scaling and Elastic Load Balancing with examples

Auto Scaling:
• Automatically adjusts the number of EC2 instances based on demand.
• Ensures sufficient capacity during peak load and reduces cost in low load.

Components:
• Launch template: defines AMI, instance type, security group.
• Auto Scaling Group (ASG): manages minimum, maximum, and desired number of instances.
• Scaling policies:
– Target tracking (e.g., keep CPU at 50%)
– Step scaling
– Scheduled scaling

Example:
An e-commerce site increases traffic during festivals. Auto Scaling adds more web server instances to handle the load and terminates unused instances after peak hours.

Elastic Load Balancing (ELB):
• Distributes incoming traffic across multiple EC2 instances.
• Detects unhealthy instances and routes traffic only to healthy ones.

Types:
• Application Load Balancer (ALB) – HTTP/HTTPS, layer 7.
• Network Load Balancer (NLB) – TCP/UDP, ultra-low latency.
• Classic Load Balancer – older version.

Example:
A website with multiple EC2 instances behind an ALB receives user requests. If one instance fails, ALB routes traffic to others, maintaining availability.

Auto Scaling + ELB:
When Auto Scaling adds new instances, ELB automatically registers them. When instances terminate, ELB deregisters them.

5. EBS-Backed vs Instance-Store-Backed Instances

EBS-Backed Instances:
• Root volume stored on EBS (Elastic Block Store).
• Persistent storage: data persists even after instance stops.
• Instances can be stopped and restarted without data loss.
• Snapshots allow backup and AMI creation.
• Suitable for production workloads.

Instance-Store-Backed Instances:
• Root volume stored on physical disks attached to the host.
• Non-persistent: data is lost if instance stops, fails, or host hardware changes.
• Faster I/O but temporary.
• Cannot stop and start; only reboot or terminate.
• Suitable for temporary storage, caching, buffers.

Key difference:
• EBS = persistent and reliable.
• Instance Store = temporary, high-performance but volatile.

UNIT 3

1. AWS Elastic Beanstalk and how it helps in deploying scalable apps

Elastic Beanstalk is a managed platform service that automates deployment, scaling, monitoring, and management of web applications. It removes the need to manually provision EC2, load balancers, security groups, auto scaling, and software configurations.

Core functions:
• Supports multiple languages: Java, Python, Node.js, PHP, .NET, Go, Ruby.
• Automatically provisions infrastructure: EC2, ELB, Auto Scaling Groups, RDS (optional), networking.
• Provides version control of application deployments.
• Offers environment types: Web Server Environment and Worker Environment.
• Applies platform updates through managed updates.

How it enables scalability:
• Auto Scaling is integrated and adjusts instance count based on load.
• Load balancers distribute traffic automatically.
• Developers deploy code; Beanstalk handles capacity and health monitoring.

Example:
Deploying a Node.js web app: upload ZIP file → Beanstalk creates EC2 instances, configures Nginx, sets scaling policies → app scales automatically under traffic spikes.

2. CloudFormation and CloudWatch (metrics + alarms)

AWS CloudFormation:
• Infrastructure-as-Code (IaC) service.
• Defines AWS resources in JSON/YAML templates.
• Automates creation, updating, and deletion of entire stacks in a controlled manner.
• Ensures consistent and repeatable deployments.
• Supports versioning, rollback, and drift detection.

Example:
A CloudFormation template defines VPC, subnets, EC2, RDS. Deploying the template creates the full architecture without manual configuration.

AWS CloudWatch:
• Monitoring and observability service.
• Collects metrics, logs, and events from AWS services and applications.

CloudWatch Metrics:
• CPUUtilization, NetworkIn/Out for EC2
• RequestCount, Latency for ELB
• ReadIOPS/WriteIOPS for EBS
• Invocations/Errors for Lambda
• Custom metrics from applications

CloudWatch Alarms:
• Triggered when a metric breaches a defined threshold.
• Actions include sending SNS notifications, auto scaling adjustments, or EC2 recovery.

Example:
Set an alarm: EC2 CPU > 80% for 5 minutes → trigger Auto Scaling to add an instance.

3. SNS vs SQS vs SES with examples

Amazon SNS (Simple Notification Service):
• Pub/Sub messaging system.
• Pushes messages to subscribers immediately.
• Supports protocols: SMS, email, HTTP endpoint, Lambda.
• Used for broadcast notifications.

Example:
A CloudWatch alarm triggers SNS → sends SMS to admin → triggers Lambda for automated response.

Amazon SQS (Simple Queue Service):
• Fully managed message queue.
• Decouples distributed systems.
• Messages are pulled by consumers, not pushed.
• Ensures reliable delivery, supports FIFO and standard queues.

Example:
Web server sends jobs to SQS → worker processes consume jobs → handles asynchronous processing like image resizing.

Amazon SES (Simple Email Service):
• Scalable email sending service.
• Used for transactional and marketing emails.
• Supports SMTP and API-based sending.

Example:
Sending password reset emails or bulk promotional emails.

Key differences:
• SNS → publish/subscribe, instant push notifications.
• SQS → queue for asynchronous processing, pull mechanism.
• SES → email sending.

4. IAM—users, groups, roles, policies

IAM (Identity and Access Management) controls authentication and authorization for AWS resources.

IAM Users:
• Individual identities.
• Each user has unique credentials (password, access keys).
• Used for real people or applications needing direct access.

IAM Groups:
• Collections of users.
• Policies applied to groups affect all members.
• Example: Admins, Developers, Auditors.

IAM Roles:
• Temporary access identity with permissions.
• Used by AWS services, applications, or cross-account access.
• No long-term credentials; uses temporary session tokens.

Examples:
• EC2 instance role allowing S3 access.
• Lambda execution role.
• Cross-account roles for sharing resources.

IAM Policies:
• JSON documents defining allowed/denied actions.
• Types: AWS-managed, customer-managed, inline.
• Example policy: Allow s3:ListBucket and s3:GetObject.

Policy structure:
• Version
• Statement(s)
• Effect (Allow/Deny)
• Action
• Resource
• Condition (optional)

IAM provides least-privilege security by ensuring users/services get only necessary permissions.

5. AWS KMS and Data Security (at rest & in motion)

AWS KMS (Key Management Service):
• Managed service for creating, storing, and controlling encryption keys.
• Supports symmetric and asymmetric keys.
• Integrated with S3, EBS, RDS, Lambda, CloudTrail, and many AWS services.
• Manages key rotation, access control, auditing.

Key features:
• Customer Master Keys (CMKs)
• Envelope encryption (data keys encrypted by CMKs)
• Logging via CloudTrail

Data Security at Rest:
• Protects stored data.
• AWS offers server-side encryption using KMS-managed keys (SSE-KMS).
• EBS volumes, S3 objects, RDS databases can be encrypted automatically.
• Encryption is transparent to applications.

Example:
S3 bucket with SSE-KMS → objects stored as encrypted files using AES-256.

Data Security in Motion:
• Protects data during transfer using TLS/HTTPS.
• Secure connections to APIs, endpoints, and load balancers.
• VPN and Direct Connect with encryption ensure secure hybrid connectivity.

Example:
Client → ALB (HTTPS) → EC2 uses TLS to prevent eavesdropping or tampering.

Combined security model:
• KMS handles key lifecycle.
• Encryption at rest protects against disk theft or unauthorized access.
• Encryption in motion protects against interception during transit.

UNIT 4

1. Amazon S3 – buckets, objects, storage classes, use cases

Amazon S3 (Simple Storage Service) is object storage designed for durability, availability, and scalability. Data is stored as objects inside buckets.

Buckets:
• Logical containers that store objects.
• Globally unique names.
• Properties: versioning, lifecycle rules, encryption, access policies, logging.

Objects:
• Fundamental storage units in S3.
• Consist of data, metadata, and a unique key.
• Any file type: images, videos, logs, backups.

Storage Classes:
• S3 Standard: high availability, frequent access.
• S3 Intelligent-Tiering: auto-moves data between tiers based on access patterns.
• S3 Standard-IA (Infrequent Access): lower cost, retrieval fee.
• S3 One Zone-IA: stored in one AZ.
• S3 Glacier Instant Retrieval: archival with quick access.
• S3 Glacier Flexible Retrieval: minutes to hours restore.
• S3 Glacier Deep Archive: lowest-cost long-term archival.

Use Cases:
• Website hosting (static sites).
• Backup and restore.
• Data lake storage for analytics.
• Application logs.
• Media storage and delivery.
• ML training datasets.
• Disaster recovery.

2. Difference between S3, EBS, and Glacier (type, use, cost)

Amazon S3:
• Storage type: object storage.
• Use: static files, backups, media, logs, data lakes.
• Cost: low, pay per GB stored + request charges.
• Access: HTTP-based API.

EBS (Elastic Block Store):
• Storage type: block storage.
• Use: EC2 root volumes, databases, file systems.
• Cost: higher than S3; charged per provisioned GB + IOPS.
• Access: attaches to EC2 like a virtual hard disk.

Glacier:
• Storage type: archival object storage.
• Use: long-term archival, compliance data.
• Cost: extremely low storage cost; retrieval cost/time applies.
• Access: asynchronous retrieval.

Comparison Summary:
• S3 → general-purpose object storage.
• EBS → persistent storage for running EC2 workloads.
• Glacier → cold archival with long retrieval times.

3. EBS volume types and usage

EBS provides persistent block storage for EC2 instances. Volume types:

General Purpose SSD (gp2, gp3):
• Balanced price/performance.
• Suitable for boot volumes, web servers, small databases.

Provisioned IOPS SSD (io1, io2):
• High IOPS for latency-sensitive workloads.
• Suitable for large relational DBs, NoSQL DBs, OLTP workloads.

Throughput-Optimized HDD (st1):
• High throughput, low cost.
• Suitable for big data, log processing, streaming workloads.

Cold HDD (sc1):
• Lowest-cost HDD option.
• Suitable for infrequently accessed data.

Key characteristics:
• Volumes are AZ-specific.
• Snapshots stored in S3 for backups.
• Can resize volumes without downtime.

4. DynamoDB and key features (partitioning, throughput)

DynamoDB is a fully managed NoSQL key-value and document database.

Key features:
• Serverless architecture with automatic scaling.
• Fast performance with single-digit millisecond latency.
• Multi-region replication.
• Integrated security, backups, caching with DAX.

Partitioning:
• DynamoDB distributes data across partitions based on partition key.
• Hot partitions are avoided by selecting high-cardinality partition keys.
• Partitions ensure horizontal scaling for large datasets.

Throughput:
• Two modes:
– Provisioned: specify Read Capacity Units (RCU) and Write Capacity Units (WCU).
– On-Demand: auto-scales based on request load.

• Auto-scaling adjusts RCU/WCU based on traffic.
• Burst capacity handles short spikes.

Additional features:
• Streams for change tracking.
• TTL for automatic item expiration.
• Global secondary indexes (GSI) and local secondary indexes (LSI) for flexible queries.

Use cases:
• Gaming leaderboards.
• IoT real-time data.
• E-commerce catalogs.
• Session stores.

Amazon RDS (Relational Database Service) is a managed service for relational databases.

Supported engines:

UNIT 5

1. VPC architecture – subnets, routing, ACLs, Security Groups, IGW

A Virtual Private Cloud (VPC) is an isolated, customizable virtual network inside AWS. It contains networking components that define how resources communicate internally and with the internet.

Core components:

Subnets:
• Logical divisions of the VPC’s IP address range.
• Public subnets: connected to the internet through an Internet Gateway.
• Private subnets: no direct internet access; often used for databases or internal services.
• Each subnet belongs to a single Availability Zone.

Route Tables:
• Define routing rules for outbound traffic.
• Public subnet route table includes a default route (0.0.0.0/0) pointing to IGW.
• Private subnet route table might route traffic to NAT Gateway or stay internal.

Network ACLs (NACLs):
• Stateless firewall at the subnet level.
• Controls inbound and outbound traffic.
• Rules allow or deny traffic based on IP, port, and protocol.

Security Groups:
• Stateful firewalls attached to EC2 or other resources.
• Control inbound and outbound traffic at instance level.
• Automatically allow return traffic.

Internet Gateway (IGW):
• Enables communication between VPC resources and the internet.
• Must be attached to VPC and referenced in route table for public subnets.

How the components work together:
• Public subnets allow public-facing services (web servers).
• Private subnets isolate sensitive resources.
• Route tables determine where traffic goes.
• SGs and NACLs enforce security at different layers.
• IGW enables internet connectivity.

2. Difference between Security Groups and Network ACLs

Security Groups (SG):
• Applied at instance level.
• Stateful: return traffic automatically allowed.
• Only allow rules; no explicit deny rules.
• Default: all outbound allowed, inbound denied.
• Easier to manage for applications.

Network ACLs (NACL):
• Applied at subnet level.
• Stateless: return traffic must be explicitly allowed.
• Supports both allow and deny rules.
• Default NACL allows all traffic; custom NACL denies all until configured.
• Used for broad subnet-level traffic control.

Comparison summary:
• SG → fine-grained, instance-level security.
• NACL → coarse, subnet-level traffic filtering.
• SG = stateful, NACL = stateless.

3. 3-tier VPC architecture (public & private subnets)

A 3-tier architecture separates application layers for scalability, security, and maintainability:

Tier 1: Presentation Layer (Web Tier)
• Located in public subnets.
• Contains web servers or load balancers receiving internet traffic.
• Public subnet routes traffic to IGW.

Tier 2: Application Layer (App Tier)
• Located in private subnets.
• Contains backend application servers.
• Communicates with web tier using internal IPs.
• No direct internet access.

Tier 3: Database Layer (DB Tier)
• Located in isolated private subnets.
• Contains RDS, DynamoDB (via VPC endpoints), or EC2 databases.
• No internet access; reachable only from app tier.

Traffic flow:
User → Internet → Load Balancer (public subnet) → App Servers (private subnet) → Database (private isolated subnet).

Advantages:
• Strong security isolation.
• Independent scaling of each tier.
• Fine-grained control over network access.

4. Route 53 and CloudFront with use cases

Amazon Route 53:
• Highly available DNS (Domain Name System) service.
• Functions: domain registration, DNS routing, and health checking.
• Supports routing policies:
– Simple
– Weighted
– Latency-based
– Geolocation
– Failover
– Multi-value answer

Use cases:
• Routing global user traffic to nearest server.
• Failover DNS for high-availability apps.
• Domain management for websites.

CloudFront:
• Content Delivery Network (CDN).
• Caches content at edge locations worldwide for low latency.
• Works with S3, EC2, ALB, Media Services, and custom origins.
• Provides HTTPS termination and DDoS protection.

Use cases:
• Delivering static websites stored in S3.
• Accelerating media streaming and downloads.
• Protecting applications behind AWS Shield + CloudFront.
• Reducing latency for global applications.

Together:
Route 53 directs users to CloudFront → CloudFront serves cached content or fetches from origin.

5. VPN and Customer Gateway setup

A VPN (Virtual Private Network) provides secure communication between on-premises data center and AWS VPC using encrypted tunnels over the public internet.

Components:

Customer Gateway (CGW):
• On-premises device (router or firewall) or software appliance.
• Represents your physical network in AWS.

Virtual Private Gateway (VGW):
• AWS-side gateway attached to the VPC.
• Establishes IPSec tunnel with CGW.

VPN Setup Process:

Step 1: Create a Virtual Private Gateway
Attach it to the target VPC.

Step 2: Create a Customer Gateway
Specify:
• Public IP of on-prem router
• Routing type (static or BGP dynamic routing)

Step 3: Create a VPN Connection
Link CGW and VGW.
AWS generates tunnel configuration files.

Step 4: Configure On-Premises Router
Apply tunnel configuration downloaded from AWS (supports Cisco, Juniper, etc.).

Step 5: Test Connectivity
Ping EC2 in private subnets using private IPs.

Use cases:
• Hybrid cloud setup.
• Secure connection from office network to AWS.
• Backup link for Direct Connect.
