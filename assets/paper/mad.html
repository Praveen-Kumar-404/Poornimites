Mobile Application Development

1. Android System Architecture with diagram labels.

Five-layer stack that manages execution, security, hardware access, and application behavior.

Layer 1 — Applications
 All system and user-installed apps. Examples: Phone, Contacts, Play Store, third-party apps. These use high-level APIs from the framework layer.

Layer 2 — Application Framework
 Java/Kotlin APIs that expose system services. Core managers:

  • ActivityManager
  • WindowManager
  • PackageManager
  • NotificationManager
  • LocationManager
  • ContentResolver
    This layer allows apps to access hardware, resources, and system features.

Layer 3 — Android Runtime (ART)
 Executes application bytecode. Contains:

  • ART virtual machine
  • Core Java libraries
  • AOT + JIT compiler
  • Garbage collector
     Handles memory management, thread scheduling, and performance optimization.

Layer 4 — Platform Libraries
 Native C/C++ libraries. Provide essential low-level functionality.
Examples:

  • OpenGL ES (graphics)
  • WebKit (browser)
  • SQLite (database)
  • SSL (security)
  • Media framework (audio/video)
  • Surface Manager (UI compositing)

Layer 5 — Linux Kernel
 Base of the OS. Manages:

  • Process management
  • Memory management
  • Networking
  • Power management
  • Security
  • Hardware drivers (display, audio, camera, sensors)

This stack ensures stable, secure, and efficient app execution across hardware.

2. Layers of Android: Applications, Application Framework, Android Runtime (ART), Platform Libraries, Linux Kernel.

Applications
 Topmost layer. All apps installed in the system. They interact with user and consume APIs from lower layers.

Application Framework
 Provides ready-to-use Java APIs. Enables reuse of system components like Activities, Services, Notifications, Telephony, Resources.

Android Runtime (ART)
 Runs app code. Converts .dex bytecode to machine code. Optimizes performance and memory. Includes garbage collection.

Platform Libraries
 Native layer offering core functionalities. Media playback, graphics rendering, database support, SSL, UI rendering.

Linux Kernel
 Hardware interaction and low-level management. Supplies drivers and ensures security via process isolation.

These layers work together: apps → framework → runtime/libraries → kernel → hardware.

3. ART vs Dalvik (DVM) comparison.

Dalvik Virtual Machine (DVM)
 Used in Android versions before 5.0.

  • Register-based VM.
  • Executes .dex bytecode.
  • Relies on JIT compilation only.
  • Slower execution due to repeated runtime compilation.
  • Lower memory efficiency.
  • Faster app installation (because no pre-compilation).

Android Runtime (ART)
 Replaced DVM from Android 5.0 onward.

  • Uses AOT compilation during installation and JIT during execution.
  • Generates native machine code → faster startup and execution.
  • Improved garbage collection (concurrent, low-pause).
  • Better memory allocation and reduced fragmentation.
  • Slightly longer installation time because of AOT.
  • Lower CPU usage during runtime because most code is already compiled.

4. AOT (Ahead-of-Time) compilation explanation.

AOT converts an app’s .dex bytecode into native machine code during installation, before the app ever runs.

Process

  1. User installs the app (APK).
  2. ART compiler converts bytecode → native code.
  3. Native code is stored in the /data partition.
  4. When the app launches, no major compilation is required.

Advantages

  • Faster app launch and runtime performance because most code is already compiled.
  • Reduced CPU load during execution.
  • Better battery efficiency because no repeated JIT compilation during usage.

Disadvantages

  • App takes longer to install.
  • More storage used due to compiled native binaries.

AOT is one of the core optimizations introduced in ART.

5. JIT (Just-in-Time) compilation explanation.

JIT compiles bytecode while the app is running, only for the parts of code being used frequently.

Process

  1. App launches using interpreted or baseline compiled code.
  2. ART monitors hot paths (frequently executed code).
  3. These hot paths are compiled to native code at runtime.
  4. Compiled code is cached and reused during the session.

Advantages

  • Faster installation (compared to full AOT).
  • Lower storage usage.
  • Adaptive optimization: only needed code is compiled.

Disadvantages

  • Slight runtime overhead due to compilation during execution.
  • Higher CPU usage than AOT.
  • Can cause short pauses when compiling during execution (ART reduces this with better GC and profiling).

Modern Android uses AOT + JIT hybrid to balance performance and installation speed.

6. Memory efficiency, performance, and installation differences between ART and DVM.

Memory Efficiency

DVM

  • Heavier GC pauses.
  • Frequent JIT compilations consume RAM and CPU.
  • Higher memory fragmentation.

ART

  • Improved garbage collectors (Concurrent, Generational).
  • Optimized object allocation.
  • Reduces memory churn and fragmentation.
  • Better handling of large heaps.

Result: ART provides significantly better memory efficiency.

Performance

DVM

  • Slower execution because everything is compiled at runtime (JIT only).
  • More CPU usage due to repeated compilation.

ART

  • Faster launch times (AOT pre-compilation).
  • Faster code execution (native binaries).
  • Less CPU usage → improved performance and battery life.

Result: ART delivers higher runtime performance.

App Installation Process

DVM Installation

  • Quick installation.
  • No pre-compilation.
  • Runtime JIT handles execution.

ART Installation

  • Longer installation time because AOT compiles bytecode → native code.
  • More storage required for native artifacts.
  • But runtime becomes significantly faster.

Result: Install slow → run fast under ART.

7. Android UI/UX principles: Clarity, Consistency, Feedback, Efficiency, Navigation, Accessibility.

Framework for designing interfaces that are clear, predictable, accessible, and efficient.

Clarity
 UI must communicate information without confusion.

  • Text must be readable (proper contrast, spacing, font size).
  • Icons must be universally recognizable.
  • Avoid clutter. Each screen performs one primary task.

Consistency
 Users learn patterns. Reusing layouts, colors, and controls builds familiarity.

  • Follow Material Design guidelines.
  • Keep buttons, spacing, and navigation uniform across screens.
  • Consistent naming and behavior (e.g., Back button always works the same way).

Feedback
 Every user action must produce an immediate system response.

  • Button ripple animation, progress bars, loaders.
  • Error messages for incorrect input.
  • Disabled states for unavailable actions.

Efficiency
 Reduce user effort and steps required to accomplish tasks.

  • Autofill, auto-suggestions, shortcuts.
  • Clear CTAs (Call to Action).
  • Minimize data entry with defaults and templates.

Navigation
 Information must be organized logically with predictable movement.

  • Bottom navigation bar for 3–5 primary pages.
  • Drawer menu for secondary options.
  • Use fragments for multi-screen flows.

Accessibility
 Ensure usability for all users, including those with disabilities.

  • Large touch targets (48dp).
  • Voice navigation compatibility (TalkBack).
  • Proper color contrast and readable text.
  • Alternative text for images (contentDescription).

These principles ensure predictable, responsive, and inclusive user interfaces.

8. MVP architecture: roles of Model, View, and Presenter.

MVP is a UI architecture pattern that separates responsibilities into three layers.

Model

  • Manages data, business logic, API calls, database operations, shared preferences.
  • Returns data to Presenter (not the View).
  • Does not know anything about UI.

View

  • Activities, Fragments, or XML-based UI.
  • Displays data to the user.
  • Forwards user actions (clicks, inputs) to Presenter.
  • Contains zero business logic.

Presenter

  • The “middle controller.”
  • Receives user actions from the View.
  • Requests data from the Model.
  • Applies logic and updates the View with results.
  • Manages UI states (loading, errors, data display).
  • Has no Android UI code; communicates via interface.

Flow
 View → Presenter → Model
Model → Presenter → View

MVP improves testability, reduces UI complexity, and isolates business logic from the UI layer.

9. Reason long-running operations cannot run on the Main/UI thread.

The Main/UI thread performs:

  • Rendering of layouts
  • Handling user input
  • Updating the screen at ~16ms/frame
  • Running Activity/Fragment lifecycle callbacks

If heavy or blocking tasks run on this thread, it freezes the UI.

Examples of long-running operations

  • Network requests
  • File read/write
  • Database queries
  • Image processing
  • Loops with large computations

Consequences

  1. Dropped frames → laggy interface
  2. Frozen screen → user cannot interact
  3. ANR (Application Not Responding) error
    Triggered when main thread is blocked for     >5 seconds.

System message:
 “App isn’t responding. Do you want to close it?”

To avoid blocking, long-running tasks must use:

  • AsyncTask (older)
  • Thread / ExecutorService
  • HandlerThread
  • WorkManager
  • Coroutine (Kotlin)
  • RxJava
  • LiveData with ViewModel
  • Background Services

The UI thread must remain non-blocking to maintain smooth performance and avoid system-level ANR termination.

10. ANR (Application Not Responding): cause and prevention.

Definition
 ANR occurs when the Main/UI thread is blocked for too long.
Android expects the UI thread to update smoothly every ~16ms. If it stops responding:

  • 5 seconds of blocked UI thread → ANR dialog
  • 10 seconds of blocked BroadcastReceiver → ANR
  • 20 seconds of blocked Service → ANR

Causes

  1. Long operations on the main thread:
  • Network calls
  • Large database queries
  • File I/O
  • Heavy loops, sorting, JSON parsing
  • Bitmap decoding or image processing
  1. Deadlocks
    Two threads waiting on each other, blocking the UI thread.
  2. Synchronous functions that take too long
    Example: Thread.sleep(5000) on the main thread.
  3. Blocking UI during lifecycle methods
    If onCreate(), onResume(), or onTouchEvent() run heavy code → ANR.

Prevention

  • Move heavy operations to background threads.
  • Never use network calls directly on the main thread.
  • Use WorkManager, Coroutines, Executors, or AsyncTask (older method).
  • Use loaders or Room’s asynchronous queries.
  • Keep UI thread only for rendering and event handling.
  • Break large tasks into smaller chunks.

ANR avoidance depends entirely on keeping the UI thread non-blocking.

11. AsyncTask explanation.

AsyncTask is a utility (now deprecated, but still used in curriculum) to run short background operations and update the UI without manually handling threads.

Purpose

  • Perform background work
  • Update UI before, during, and after the task
  • Avoid blocking the main thread
  • Provide callbacks on UI thread automatically

How it works
 AsyncTask provides three generic parameters:
AsyncTask<Params, Progress, Result>

  • Params → Input type for background work
  • Progress → Type used for progress updates
  • Result → Output returned after execution

Thread model

  • onPreExecute(), onProgressUpdate(), and onPostExecute() → run on main thread
  • doInBackground() → runs on background thread

AsyncTask’s job is ideal for small tasks like:

  • Downloading a small file
  • Network call with small response
  • Short database operation
  • Basic image processing

Not suitable for long background work (should use WorkManager / Services).

12. AsyncTask lifecycle methods: onPreExecute(), doInBackground(), publishProgress(), onProgressUpdate(), onPostExecute().

AsyncTask has a fixed execution sequence with specific thread rules.

1. onPreExecute()

  • Runs: Main/UI Thread
  • Used for: setup, showing progress indicators
    Example: showing a ProgressBar before the task runs.

2. doInBackground(Params…)

  • Runs: Background Thread
  • Heavy operations executed here
  • Must not update the UI directly
  • Can call publishProgress() to update UI

Example tasks:

  • Network operations
  • File download
  • Database read/write
  • Long computation

3. publishProgress(Progress…)

  • Called from doInBackground()
  • Triggers onProgressUpdate()

4. onProgressUpdate(Progress…)

  • Runs: Main/UI Thread
  • Updates progress to user
    Examples:
  • Updating a progress bar
  • Updating loading text

5. onPostExecute(Result)

  • Runs: Main/UI Thread
  • Receives the result returned by doInBackground()
  • Used for final UI updates, hiding progress bar, showing results

Complete flow
 onPreExecute() → doInBackground() → publishProgress() → onProgressUpdate() → onPostExecute()

Thread summary

  • UI thread: onPreExecute, onProgressUpdate, onPostExecute
  • Background thread: doInBackground

AsyncTask ensures UI remains responsive while background work is ongoing.

13. Role and structure of Content Providers.

Role
 Content Providers allow structured data to be shared between applications in Android.
They act as a data access layer with a uniform interface, regardless of where the data comes from.

Apps typically use Content Providers for:

  • Contacts
  • SMS
  • Media (Images, Audio, Video)
  • Calendar
  • Call logs

A Content Provider serves three purposes:

  1. Provide CRUD access to data.
  2. Expose data to other apps using permissions.
  3. Standardize queries using URIs and MIME types.

Structure of a Content Provider
 A Content Provider class must implement:

  • onCreate() – initialization
  • query() – read
  • insert() – create
  • update() – update
  • delete() – delete
  • getType() – return MIME type
  • Content URI definitions
  • Authority name
  • URI Matcher to identify operations

Internal data storage can be:

  • SQLite database
  • Files
  • SharedPreferences
  • Network API (rare)

Content Providers standardize data access and enforce security boundaries.

14. CRUD operations in Content Providers: insert, query, update, delete.

Content Providers expose four major database-style operations:

1. insert() — Create
 Used to add a new record.
Receives: URI + ContentValues
Returns: new record’s URI

Example: adding a new contact.

2. query() — Read
 Used to retrieve data.
Receives: URI, projection, selection, selectionArgs, sortOrder
Returns: Cursor pointing to result set.

Example: reading all images from gallery.

3. update() — Update
 Used to modify existing data.
Receives: URI, ContentValues, selection, selectionArgs
Returns: number of rows updated.

Example: editing a saved contact name.

4. delete() — Delete
 Used to remove rows.
Receives: URI, selection, selectionArgs
Returns: number of rows deleted.

Example: deleting SMS messages.

These CRUD functions provide a standardized interface for any app to interact with the provider.

15. Content URI components: prefix, authority, path, ID.

A Content URI uniquely identifies data in a Content Provider.

General format:

content://authority/path/id

Breakdown:

1. Prefix: content://
 Indicates the URI belongs to a Content Provider category.

2. Authority
 Unique name of the Content Provider.
Examples:

  • com.android.contacts
  • com.android.calendar
  • com.example.myprovider

Acts like a domain name for the provider.

3. Path
 Represents a table or collection.
Examples:

  • contacts
  • users
  • students
  • images/media

Can have multiple levels:
path1/path2

4. ID
 Represents a single record.
Example:
content://contacts/people/5 → contact with ID = 5

ID is optional.
If omitted → refers to whole table.

Content URI Examples
 Contacts list:
content://com.android.contacts/contacts

Specific contact:
content://com.android.contacts/contacts/3

Images:
content://media/internal/images/media

This structure allows external apps to access exactly the data they need while preserving control and security.

16. Function of ContentResolver and CursorLoader.

ContentResolver
 ContentResolver is the client-side API that an app uses to communicate with any Content Provider.

It acts as a middle layer between your app and the provider.

Responsibilities

  • Sends CRUD requests to the Content Provider.
  • Converts your method calls into provider operations.
  • Ensures thread-safe access to data.
  • Handles URI routing (which provider to query).
  • Returns a Cursor when reading data.

Key methods

  • insert(Uri uri, ContentValues values)
  • query(Uri uri, String[] projection, String selection, String[] args, String sortOrder)
  • update(Uri uri, ContentValues values, String selection, String[] args)
  • delete(Uri uri, String selection, String[] args)
  • getType(Uri uri)

ContentResolver does not store data; it only forwards requests.

CursorLoader

CursorLoader loads data from a Content Provider asynchronously.

Purpose

  • Avoids performing queries on the UI thread.
  • Automatically manages the Cursor lifecycle.
  • Reloads data when the underlying data changes.

How it works

  • Runs queries in the background thread.
  • Returns a Cursor to the UI thread when ready.
  • Automatically updates the data if the provider notifies changes.

CursorLoader is essential for smooth UI, especially when loading large datasets like contacts or media files.

17. How an app queries data from a Content Provider.

The data query process has a fixed pipeline:

Step 1 — Identify the Content URI
 Example (contacts list):
content://com.android.contacts/contacts

Step 2 — Get ContentResolver instance
 Usually via:
ContentResolver resolver = getContentResolver();

Step 3 — Call query()
 App requests data through ContentResolver:

Cursor cursor = resolver.query(

    uri,

    projection,

    selection,

    selectionArgs,

    sortOrder

);

Step 4 — ContentResolver forwards request
 ContentResolver routes the query to the correct Content Provider via URI matching.

Step 5 — Content Provider executes the query
 Provider executes:

  • an SQLite query
  • a file read
  • or any custom data source

Step 6 — Provider returns a Cursor
 The Cursor points to the rows returned.

Step 7 — App reads data from Cursor
 Typical methods:

  • cursor.moveToFirst()
  • cursor.getString(columnIndex)
  • cursor.getInt(columnIndex)
    Native

Step 8 — Close Cursor
 cursor.close()
 Prevents memory leaks.

If using CursorLoader
 Steps 3–8 are done automatically in background threads.

18.  vs Hybrid vs Cross-Platform apps based on: performance, UI, rendering, architecture, development workflow.

Native Apps
 Built using platform’s official languages and SDKs.

  • Android → Java/Kotlin
  • iOS → Swift/Objective-C

Performance
 Fastest, uses native machine code.

UI
 Perfect platform look and feel.

 Uses system UI components (Views, UIKit, etc.).

 Tight integration with OS APIs.

 Separate codebases for Android and iOS → higher cost.

 Built using web technologies (HTML/CSS/JS) and run inside a WebView.
Frameworks: Cordova, Ionic.

Performance
 Slower due to WebView rendering.

 Not fully native; web-based.
May look inconsistent with system UI.

 HTML/CSS rendered inside a browser container.

Architecture
 Plugins used to access hardware.

Development Workflow
 Single codebase → easy and cheap.

 Frameworks: Flutter, React Native, Xamarin.

Performance
 Near-native (Flutter best among them).

 Custom widgets (Flutter), cross-platform components.

 Uses a dedicated rendering engine (Flutter uses Skia).

Architecture
 Shared UI + shared logic; compiled for both platforms.

Development Workflow
 Single codebase → fast development + good performance balance.

19. How Flutter’s Skia rendering engine works.

Flutter does not use Android’s native UI components (TextView, Button, etc.).
Instead, Flutter draws its entire UI using the Skia rendering engine, the same engine used by Google Chrome.

Key Principles

A. Flutter controls every pixel
 UI = drawn frame-by-frame on a canvas.
There is no dependency on OEM widgets.
This ensures identical UI on Android, iOS, Web, and Desktop.

B. Rendering pipeline
 Flow of UI creation:

  1. Dart Code
     Your widgets and layout logic.
  2. Flutter Framework
     Builds a widget tree → element tree → render tree.
  3. Rendering Layer
     RenderObjects calculate layout and positioning.
  4. Skia Engine
     Draws shapes, text, images on the GPU.
  5. Hardware/OS
     Displays the final frame at ~60–120 FPS.

C. Advantages of Skia Rendering

  • Same UI on all platforms (no platform-specific inconsistencies).
  • Very high FPS (GPU accelerated).
  • Faster animations than native in many cases because Flutter bypasses OEM UI layers.
  • No need for bridges like React Native → eliminates performance bottlenecks.

D. Result
 Flutter apps feel smooth, responsive, and consistent because the whole UI is drawn by Skia instead of depending on slow platform rendering paths.

20. How Hot Reload Improves Developer Productivity.

Hot Reload injects updated Dart code directly into the running app without restarting the app and without losing the current UI state.

What it updates

  • UI changes
  • Layout changes
  • Minor logic changes
  • Widget updates

What stays preserved

  • Variables
  • Scroll position
  • Navigation stack
  • Form inputs
  • App state

Why it speeds up development

  • No app rebuild time.
  • No state loss → developer continues exactly where they left off.
  • UI changes appear instantly (under 1 second).
  • Rapid trial-and-error design.

Typical use case

Fixing UI padding, spacing, alignment → results appear immediately.

21. How Hot Restart Improves Development Speed.

Hot Restart does not inject code into the current Dart VM.
It restarts the Dart VM, clearing all state, but still avoids a full app recompilation.

What it updates

  • Full code changes
  • Updated app logic
  • Initialization logic
  • Build methods
  • Main() or global variables

What is reset

  • App state
  • Navigation
  • Variables
  • User session

Why Hot Restart helps

  • Much faster than a full rebuild/install cycle.
  • Required when Hot Reload cannot apply changes (e.g., structural changes).

Examples of changes requiring Hot Restart:

  • Changing main() method
  • Modifying global variables
  • Changing app-level state models
  • Adding or removing dependencies
