1. Java Threading and States

Ways to Create a Thread

a) Extending Thread class
 A subclass of Thread overrides the run() method and starts with start().

class MyThread extends Thread {

    public void run() {

        System.out.println("Thread running");

    }

}

new MyThread().start();

b) Implementing Runnable interface
 Implements Runnable and passes the object to a Thread instance.

class MyRunnable implements Runnable {

    public void run() {

        System.out.println("Runnable running");

    }

}

new Thread(new MyRunnable()).start();

Preferred Approach: Runnable interface

  • Allows class inheritance from another parent.
  • Promotes clean separation between task logic and thread control.
  • Encouraged in large, modular Android apps.

Purpose of run() Method

Defines the code that executes when a thread starts.
Executed automatically when start() is called, not when run() is called directly.

Function of sleep(long millis)

Temporarily suspends a thread for a defined time.

  • Thread remains alive, not terminated.
  • Used to control execution timing or wait for external resources.

Thread.sleep(1000); // sleeps for 1 second

2. AsyncTask in Android

Concept

AsyncTask is used to perform background operations and publish results to the UI thread without manually handling threads.
It helps maintain smooth UI performance by offloading heavy tasks (networking, file I/O, etc.).

Lifecycle Methods

  1. onPreExecute() – Runs on the main thread before background execution (setup, show progress).
  2. doInBackground(Params...) – Executes in the background thread. Used for long-running logic.
  3. onProgressUpdate(Progress...) – Updates UI about progress.
  4. onPostExecute(Result) – Executes on main thread after completion; displays result.

Execution Example

class MyTask extends AsyncTask<Void, Void, String> {

    protected void onPreExecute() {

        // setup

    }

    protected String doInBackground(Void... params) {

        return "Task Completed";

    }

    protected void onPostExecute(String result) {

        // update UI

        System.out.println(result);

    }

}

new MyTask().execute();

Advantages

  • Simplifies background threading.
  • Easy UI synchronization.
  • Prevents “Application Not Responding” (ANR) errors.

Limitation

  • Deprecated in newer Android versions.
  • Replaced by WorkManager, ExecutorService, or Kotlin Coroutines for better control and lifecycle handling.

3. Services and Background Operations

Concept of Services

A Service in Android is an application component that performs background operations without a user interface.
Used for continuous or long-running tasks like music playback, data sync, or file download.
Services continue to run even if the user switches to another app.

Types of Services

1. Unbound Service

  • Started using startService() or ContextCompat.startForegroundService().
  • Works independently of the activity.
  • Continues until it stops itself using stopSelf() or the system stops it.

Example:

Intent intent = new Intent(this, MusicService.class);

startService(intent);

 public class MusicService extends Service {

    public int onStartCommand(Intent intent, int flags, int startId) {

        // play music

        return START_STICKY;

    }

    public IBinder onBind(Intent intent) { return null; }

}

2. Bound Service

  • Created using bindService().
  • Provides an interface for client components (like Activities) to interact with it.
  • Runs only as long as another component is bound.

Example:

Intent intent = new Intent(this, DownloadService.class);

bindService(intent, connection, Context.BIND_AUTO_CREATE);

 public class DownloadService extends Service {

    private final IBinder binder = new LocalBinder();

    public class LocalBinder extends Binder {

        DownloadService getService() { return DownloadService.this; }

    }

    public IBinder onBind(Intent intent) { return binder; }

}

Notifications in Background Services

When a service runs for a long time or in the foreground, Android requires a notification to inform users.
Implemented using NotificationManager with an ongoing notification.

Example:

Notification notification = new NotificationCompat.Builder(this, "channel_id")

    .setContentTitle("File Upload")

    .setContentText("Uploading in background...")

    .setSmallIcon(R.drawable.ic_upload)

    .build();

startForeground(1, notification);

Purpose

  • Keeps tasks active without blocking UI.
  • Ensures user awareness of long-running operations.
  • Central to background task scheduling, media apps, and data sync.

4. Content Providers and Resolvers

Concept of Content Providers

A Content Provider manages shared app data.
It allows multiple applications to access, read, or modify structured data (like contacts, media, or custom databases) using a uniform URI system.

  • Acts as a layer between app data storage and external apps.
  • Data can come from SQLite, files, or other sources.
  • Example built-in providers: ContactsProvider, MediaStore, CallLog.

Structure of Content URI

Format:

content://authority/path/id

Components:

  • Scheme: Always content://
  • Authority: Unique name identifying the provider (e.g., contacts or your app’s package).
  • Path: Specifies which table or dataset to access.
  • ID: Optional, specifies a single record.

Example:
 content://com.android.contacts/contacts/5

Purpose of Path Component

  • Defines the table or type of data to query.
  • Example: contacts path in URI gives access to the entire contacts table.

Core Components for Access

  1. ContentProvider: Defines data access logic (query(), insert(), update(), delete()).
  2. ContentResolver: Handles communication between app and provider.
  3. Content URI: Identifies data to be accessed.

ContentResolver Object

Acts as a bridge between the app and the ContentProvider.
It sends requests and receives data responses.

Example:

Cursor cursor = getContentResolver().query(

    ContactsContract.Contacts.CONTENT_URI,

    null, null, null, null

);

Performs CRUD operations through standardized methods:

  • query()
  • insert()
  • update()
  • delete()

Advantages

  • Secure, permission-controlled data sharing.
  • Decouples data access logic from app logic.
  • Enables other apps to access structured data through URIs.

5. Google Location Services

Overview

Google’s Fused Location Provider API (FLP) delivers the most accurate location updates using a combination of GPS, Wi-Fi, cellular networks, and sensors.
It replaces older LocationManager methods for better efficiency, lower battery use, and unified access.

Core Class: FusedLocationProviderClient

This class, provided by Google Play Services, manages all location-related requests asynchronously.
Declared as:

FusedLocationProviderClient fusedClient = LocationServices.getFusedLocationProviderClient(this);

Method to Get Last Known Location

The method used is:

getLastLocation()

  • Returns a Task<Location> object.
  • Retrieves the device’s most recent known location from cached sources.
  • Does not actively request a new GPS fix.

Example Code:

fusedClient.getLastLocation()

    .addOnSuccessListener(this, location -> {

        if (location != null) {

            double lat = location.getLatitude();

            double lon = location.getLongitude();

        }

    });

Key Notes

  • Requires runtime permissions for ACCESS_FINE_LOCATION or ACCESS_COARSE_LOCATION.
  • Returns null if no previous location is available.
  • For continuous updates, use requestLocationUpdates() instead.

Recommended API Version

  • Minimum: Google Play Services version 11.0.0
  • Recommended: Always use the latest stable version of Play Services for:
  • Enhanced accuracy
  • Power efficiency
  • Support for Android 13+ background location policies

Advantages of FusedLocationProviderClient

  • Combines multiple location sources automatically.
  • Reduces battery drain compared to GPS-only tracking.
  • Provides high-level, consistent API across devices.

6. Notifications in Android

Concept

A Notification informs the user about events in the background (e.g., new message, file download, ongoing task).
It appears in the status bar and notification shade, optionally linked to an activity or service.

Core Elements of a Notification

  1. Small Icon — Mandatory visual indicator.
  2. Title — Brief summary (e.g., “Download complete”).
  3. Text Content — Additional message detail.
  4. PendingIntent — Defines the action when the user taps the notification.
  5. Sound/Vibration/Lights — Optional alerts.
  6. Priority/Channel — Controls visibility and behavior.

Basic Example

NotificationCompat.Builder builder = new NotificationCompat.Builder(this, "channel_id")

    .setSmallIcon(R.drawable.ic_notify)

    .setContentTitle("Download Complete")

    .setContentText("Your file has been saved.")

    .setPriority(NotificationCompat.PRIORITY_HIGH);

NotificationManagerCompat notificationManager = NotificationManagerCompat.from(this);

notificationManager.notify(1, builder.build());

Notification Channels (Android 8.0+)

Every notification must belong to a channel, allowing users to control settings like sound or importance per channel.

if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.O) {

    NotificationChannel channel = new NotificationChannel(

        "channel_id",

        "Downloads",

        NotificationManager.IMPORTANCE_DEFAULT

    );

    NotificationManager manager = getSystemService(NotificationManager.class);

    manager.createNotificationChannel(channel);

}

Types of Notifications

Type

Description

Simple Notification

Basic message (title, text, icon)

Expandable Notification

Displays more text or images

Heads-Up Notification

Pops up over active screen (urgent)

Progress Notification

Shows ongoing progress (e.g., download)

Foreground Notification

Tied to active background service

Key Concepts

  • Managed by the NotificationManager system service.
  • Can be triggered by a Service, Alarm, or BroadcastReceiver.
  • Dismissed manually by the user or programmatically using cancel() or cancelAll().

7. Data Storage and Management

Overview

Android provides multiple mechanisms to store and manage app data, depending on data size, sensitivity, and persistence needs.
Main options:

  • Shared Preferences (small key-value data)
  • Files (structured/unstructured data)
  • Firebase (cloud-based real-time data)

A. Shared Preferences

Used for storing small, lightweight pieces of data such as user settings, login states, or flags.

Characteristics:

  • Stores data as key-value pairs.
  • Private to the app (by default).
  • Automatically saved as an XML file in internal storage.

Code Example:

SharedPreferences sp = getSharedPreferences("MyPrefs", MODE_PRIVATE);

SharedPreferences.Editor editor = sp.edit();

editor.putString("username", "Praveen");

editor.putBoolean("isLoggedIn", true);

editor.apply();

To Retrieve:

String user = sp.getString("username", "default");

boolean logged = sp.getBoolean("isLoggedIn", false);

Use Cases:

  • Save login tokens.
  • Theme settings (dark/light mode).
  • User preferences.

B. Android File System

The Android file hierarchy separates app data from system data and provides both internal and external storage access.

File Writing Example:

String data = "Hello World";

FileOutputStream fos = openFileOutput("file.txt", MODE_PRIVATE);

fos.write(data.getBytes());

fos.close();

File Reading Example:

FileInputStream fis = openFileInput("file.txt");

int c;

while ((c = fis.read()) != -1) {

    System.out.print((char) c);

}

fis.close();

Key Notes:

  • Internal storage: private, deleted when app uninstalls.
  • External storage: requires permission, shared across apps.

C. Firebase (Cloud Storage & Database)

Firebase provides cloud-hosted services for data storage, synchronization, and authentication.
It is part of Google’s mobile development platform.

Key Features:

  1. Realtime Database — JSON-based database synced in real time.
  2. Firestore — Scalable NoSQL cloud database.
  3. Firebase Authentication — User login via email, Google, etc.
  4. Cloud Storage — Upload and serve user-generated content.
  5. Firebase Cloud Messaging (FCM) — Send notifications.

Example (Realtime Database Write):

DatabaseReference ref = FirebaseDatabase.getInstance().getReference("users");

ref.child("user1").setValue("Praveen");

Example (Read):

ref.child("user1").get().addOnSuccessListener(snapshot -> {

    String name = snapshot.getValue(String.class);

});

Advantages:

  • Cross-device data sync.
  • Offline persistence.
  • Scalable backend (no local server needed).

8. Toast Message

Concept

A Toast in Android is a small popup message that briefly appears on the screen to provide non-interactive feedback to the user.
It does not block user interaction and disappears automatically after a short duration.

Syntax

Toast.makeText(context, text, duration).show();

Parameters:

  • context → Activity or Application context.
  • text → Message to display.
  • duration → Either Toast.LENGTH_SHORT or Toast.LENGTH_LONG.

Example

Toast.makeText(getApplicationContext(), "Data Saved Successfully", Toast.LENGTH_SHORT).show();

Custom Duration Example:

Toast toast = Toast.makeText(this, "Uploading...", Toast.LENGTH_LONG);

toast.show();

Custom Layout (Optional)

You can design a custom Toast with your own XML layout.

LayoutInflater inflater = getLayoutInflater();

View layout = inflater.inflate(R.layout.custom_toast, null);

Toast toast = new Toast(getApplicationContext());

toast.setView(layout);

toast.setDuration(Toast.LENGTH_SHORT);

toast.show();

Use Cases

  • Display short success/error messages.
  • Notify user of completed background operations.
  • Give lightweight, transient feedback.
