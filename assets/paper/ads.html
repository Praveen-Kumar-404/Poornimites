Advance Data Structure

Introduction to trees

A tree is a way to organize items so relationships are clear.
Think of a family tree. One person at top. Branches go down.
Parts:

  • Node: one item (a person).
  • Root: top node.
  • Child: direct descendant.
  • Parent: direct ancestor.
  • Leaf: node with no children (end of a branch).
  • Edge: connection between parent and child.
  • Height: longest path from a node down to a leaf.
    Why use trees: fast search, organized hierarchy, clear parent-child links.

Binary Trees and Binary Search Trees (BST)

  • Binary tree: each node has 0, 1, or 2 children.
  • Binary Search Tree (BST): left child values < node value. Right child values > node value.
    BST property makes search fast. Search time depends on tree height.
    Example small BST:

  8

  / \

 3  10

/ \   \

1  6   14

Search 6: start at 8, go left to 3, go right to 6. Found.

Tree traversals (ways to visit all nodes)

  • Inorder (Left, Node, Right): visits nodes in sorted order for BST.
  • Preorder (Node, Left, Right): visits root first. Used to copy tree.
  • Postorder (Left, Right, Node): visits root last. Used to delete tree.
    Example on above BST inorder gives 1,3,6,8,10,14.

AVL Trees (self-balancing BST)

Problem with BST: can become tall and slow if nodes insert in order.
AVL trees keep height balanced. Balance factor = height(left) − height(right). Allowed values: −1, 0, +1. If outside that, tree is rebalanced using rotations.

Rotations fix imbalance. Two types: single and double.

Single Rotation (LL and RR)

Used when imbalance is in straight line.

LL case (left-heavy left child). Before:

   z

   /

  y

 /

x

After right rotation at z:

  y

  / \

 x   z

RR case symmetric: left rotation at z.

Double Rotation (LR and RL)

Used when imbalance zig-zags.

LR case (left-right): perform left rotation on left child, then right rotation on root.
Before:

 z

 /

y

 \

  x

Step 1: left rotate y -> becomes y'
Step 2: right rotate z -> becomes balanced.

Double rotations similarly for RL.

AVL operations complexity

  • Search, insert, delete: O(log n) time because height remains ~log n.
    Rotations are O(1) local operations but maintain global balance.

B-Trees (multiway balanced trees used on disk)

Purpose: store lots of keys per node so fewer disk reads are needed. Widely used in databases and filesystems.

Key ideas:

  • A B-tree node can contain many keys and many children. Parameter t (minimum degree) controls sizes.
  • Every node except root must have at least t−1 keys and at most 2t−1 keys.
  • When a node overflows (too many keys) split it and move middle key up. Splits can go up to root and increase tree height.
  • Deletion may require borrowing keys from siblings or merging nodes to keep sizes valid.

Property: height is low even for huge data sets. Search, insert, delete all O(log n), where log base depends on branching factor.

Multi-way Search Trees — 2-3 Trees

A simple balanced multiway tree. Each internal node is either:

  • 2-node: contains 1 key, has 2 children.
  • 3-node: contains 2 keys, has 3 children.

All leaves at same depth. Guarantees balanced height.

Searching in a 2-3 Tree

Start at root. Compare value against keys in node. Choose child interval that fits and move down. Repeat until leaf.

Inserting in a 2-3 Tree

  1. Insert key into leaf node.
  2. If leaf becomes a 4-node (3 keys), split it into two nodes and push middle key up to parent.
  3. If parent becomes 4-node, repeat split upward.
  4. If split reaches root, root splits and tree height increases by 1.
    Result: tree stays balanced.

Example short sequence: insert keys in leaf, split, push middle up.

Deleting from a 2-3 Tree

  1. Remove key. If removal from internal node, replace with inorder successor from leaf.
  2. If a node underflows (fewer keys than allowed), fix by:
  • Borrowing from sibling that has extra keys, moving keys through parent.
  • Merging with sibling and pulling a key down from parent.
  1. Merges can propagate up and shrink height if root becomes a single child.

Deletion rules ensure leaves stay at same depth and the tree stays balanced.

Red-Black Trees (balanced binary trees using colors)

Red-black trees are binary search trees with color rules that maintain balance with simpler local changes than AVL. They give balance close to AVL but with simpler rebalancing in many implementations.

Properties:

  1. Each node is red or black.
  2. Root is black.
  3. All NIL leaves (external nulls) are black.
  4. If a node is red, its children are black. (No two red in a row.)
  5. For every node, every path from it to descendant NIL nodes has the same number of black nodes.

These rules bound longest path to at most twice the shortest. Height is O(log n).

Rotations in red-black trees

Same left and right rotations as AVL. Rotations do not change in-order sequence. They move nodes locally to restore structure.

Insertion in red-black trees (outline)

  1. Insert new node as red like BST insertion.
  2. Fix violations with recoloring and rotations until red-black properties hold.
    Cases use parent and uncle colors to decide actions. Recolor or rotate. End with root black.

Deletion in red-black trees (outline)

  1. Remove node like in BST. If removed node or replacement is black, properties can break.
  2. Fix double-black situations by a sequence of recolorings and rotations. Cases depend on sibling color and sibling’s children.

Red-black trees are used in many libraries because they offer good worst-case time and relatively simple code paths.

Relationship between these trees

  • AVL gives stricter balance. Height smaller. Slightly more rotations. Good when searches dominate.
  • Red-black gives weaker balance. Fewer rotations on average. Good when insert/delete mix exists. Used in many language libraries.
  • 2-3 and B-trees are multiway balanced trees. 2-3 trees are simple theoretical models. B-trees scale well on disk because nodes hold many keys.
  • A red-black tree corresponds to a 2-3-4 tree in structure. Conversions exist.

Time and space summary

  • Search in balanced trees: O(log n).
  • Insert/delete in balanced trees: O(log n) with O(1) extra per rotation or split.
  • Space: O(n) for nodes and pointers.

Simple classroom examples to build intuition

  • Build a small BST with 7 numbers. Show inorder gives sorted list.
  • Insert a value that makes BST skewed. Show height increase.
  • Convert to AVL by showing rotation steps.
  • Take a 2-3 tree leaf overflow and split it. Show parent gets new key.
  • Color a tiny tree red/black and show insertion recolor/rotation steps.

Introduction to Graphs

A graph shows connections between things.
Example: cities connected by roads, or friends linked in a social network.

A graph has:

  • Vertices (nodes): points like cities or people.
  • Edges (links): connections between them (roads, friendships).

Types of graphs

  • Undirected graph: edges go both ways (like a two-way road).
  • Directed graph (digraph): edges have direction (like one-way roads).
  • Weighted graph: each edge has a number (cost, distance, time).
  • Unweighted graph: all edges are equal.

Graphs can have:

  • Paths: a sequence of edges from one vertex to another.
  • Cycles: a path that starts and ends at the same vertex.
  • Connected graph: there’s a path between every pair of vertices.

Graphs are stored using:

  • Adjacency matrix: table showing which nodes connect.
  • Adjacency list: each node keeps a list of its neighbors.

Elementary Graph Algorithms

1. Traversal — visiting all vertices

Two main methods:

  • Depth First Search (DFS): go as far as possible down one branch before backtracking.
  • Breadth First Search (BFS): explore all neighbors first before going deeper.

Both are used for searching, connectivity, and cycle detection.

2. Topological Sort

Used only for Directed Acyclic Graphs (DAGs) — graphs with direction and no cycles.

Concept

Arrange tasks in an order that respects dependencies.
Example:

  • “Cook rice” depends on “Boil water”.
  • “Eat” depends on “Cook rice”.
    So valid order: Boil water → Cook rice → Eat.

Algorithm

There are two main ways:

  • Using DFS: visit each node; when finished, add to stack; reverse stack gives order.
  • Kahn’s Algorithm (indegree method):
  1. Count how many edges point into each node (called indegree).
  2. Pick a node with indegree 0 (no dependencies).
  3. Remove it and its edges.
  4. Repeat until all nodes removed.

Use cases

  • Course prerequisite ordering.
  • Build systems (which module compiles first).
  • Task scheduling.

Single-Source Shortest Path Algorithms

Find the shortest route from one starting node to every other node.

1. Dijkstra’s Algorithm

Works when all edge weights are non-negative.

Idea

Like finding the shortest way through a road map where all distances are positive.

Steps

  1. Start at the source vertex. Distance to itself = 0. All others = infinity.
  2. Choose the vertex with the smallest known distance.
  3. Update its neighbors’ distances if shorter path found through it.
  4. Mark vertex as visited (finalized).
  5. Repeat until all vertices processed.

Example

Graph of cities:

A --5-- B --3-- C

 \      |

  2     1

   \    |

    D --4-- E

Start from A:

  • A→B = 5, A→D = 2
  • D→B = 3 (so A→D→B = 5, same)
  • Continue until all shortest paths found.

Complexity

O(V²) for simple arrays, O((V+E) log V) with a priority queue (heap).

Uses

GPS route finding, network routing.

2. Bellman–Ford Algorithm

Handles negative weights (but not negative cycles).

Idea

Relax all edges repeatedly. Each relaxation checks if a shorter path exists.

Steps

  1. Set source distance = 0, others = infinity.
  2. Repeat |V|−1 times: for every edge (u, v), if distance[v] > distance[u] + weight(u,v), update it.
  3. To detect negative cycles, do one more pass — if a distance still improves, cycle exists.

Example use

Currency exchange or graphs with both profits and losses.

Complexity

O(V × E). Slower than Dijkstra, but more general.

All-Pairs Shortest Paths

Find shortest distances between every pair of vertices.

1. Floyd–Warshall Algorithm

A dynamic programming approach. Works for both positive and negative weights (no negative cycles).

Concept

Imagine checking if using a certain vertex as an intermediate makes a path shorter.

Steps

Let dist[i][j] = current shortest distance from i to j.
For every vertex k:
For every pair (i, j):
dist[i][j] = min(dist[i][j], dist[i][k] + dist[k][j])

Example

For 4 nodes, initially fill matrix with direct edge weights.
Then gradually improve paths by checking via each intermediate node k.

Complexity

O(V³) — good for small or medium graphs.

Uses

Map systems, network routing, flight connection costs, distance tables.

Summary of Concepts

  1. Graph basics: structure to show relationships.
  2. Traversal: BFS and DFS for exploration.
  3. Topological sort: ordering of dependent tasks.
  4. Shortest path algorithms: find minimum total cost between nodes.
  • Dijkstra for positive edges.
  • Bellman–Ford for negative edges.
  • Floyd–Warshall for all pairs.
