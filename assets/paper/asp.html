(i) ConnectionsZone Control

  • The ConnectionsZone control in ASP.NET Web Parts is a specialized zone that enables users to create, modify, or delete connections between Web Parts at runtime.
  • It is used when a page has multiple Web Parts that need to communicate with each other—one acting as a provider and another as a consumer.
Purpose
  • Allows runtime connection management between Web Parts.
  • Enables data sharing between Web Parts (e.g., one Web Part providing a list of products, another displaying product details).
Working
  1. The WebPartManager control manages Web Parts and zones on the page.
  2. When the page enters connect mode, the ConnectionsZone becomes visible.
  3. It displays possible connection options that users can configure.
  4. Once a connection is made, Web Parts can share data via connection interfaces like:
  • IWebPartField – shares a single field value.
  • IWebPartTable – shares tabular data.
  • IWebPartRow – shares a single row.
Example

<asp:WebPartManager ID="WebPartManager1" runat="server" />

<asp:ConnectionsZone ID="ConnectionsZone1" runat="server" HeaderText="Connect Web Parts">

</asp:ConnectionsZone>

This setup allows users to connect Web Parts dynamically in connect mode.

Advantages
  • Promotes modular and reusable component design.
  • Enables interactive dashboards without code changes.
  • Allows end users to customize how parts of a page communicate.

(ii) Difference Between TreeView and Menu Control

Feature

TreeView Control

Menu Control

Purpose

Displays hierarchical data in a tree-like format (parent–child nodes).

Displays site navigation menus (horizontal or vertical).

Typical Use

File system navigation, XML data display.

Website navigation bar or dropdown menu.

Structure

Each node can have multiple child nodes.

Menu items can have submenus, often one level deep.

Orientation

Always vertical.

Can be horizontal or vertical.

Expansion

Supports expand/collapse functionality.

Usually hover or click dropdowns.

Bound Data Source

Often bound to SiteMapDataSource or XML file.

Commonly bound to SiteMapDataSource.

Appearance

Looks like a file explorer.

Looks like navigation tabs or dropdowns.

Example

<asp:TreeView ID="TreeView1" runat="server">

  <Nodes>

    <asp:TreeNode Text="Home" NavigateUrl="~/Default.aspx"></asp:TreeNode>

    <asp:TreeNode Text="About" NavigateUrl="~/About.aspx"></asp:TreeNode>

  </Nodes>

</asp:TreeView>

<asp:Menu ID="Menu1" runat="server">

  <Items>

    <asp:MenuItem Text="Home" NavigateUrl="~/Default.aspx" />

    <asp:MenuItem Text="About" NavigateUrl="~/About.aspx" />

  </Items>

</asp:Menu>

Key Difference Summary
  • TreeView: hierarchical display, suitable for directories or multi-level data.
  • Menu: navigational display, suitable for website menus.
  • TreeView supports expand/collapse; Menu supports dropdown.
  • TreeView is for data display; Menu is for site navigation.

(i) Write Code for Adding a Node in TreeView Control

ASP.NET Markup

<asp:TreeView ID="TreeView1" runat="server"></asp:TreeView>

C# Code-Behind

protected void Page_Load(object sender, EventArgs e)

{

    if (!IsPostBack)

    {

        // Create parent node

        TreeNode rootNode = new TreeNode("Root Node");

       

        // Create child node

        TreeNode childNode = new TreeNode("Child Node");

       

        // Add child node to root

        rootNode.ChildNodes.Add(childNode);

       

        // Add root node to TreeView

        TreeView1.Nodes.Add(rootNode);

    }

}

Explanation
  • TreeNode represents a single node.
  • ChildNodes.Add() adds a node under another node.
  • The TreeView1.Nodes.Add() adds the root node to the TreeView control.
  • This displays a simple parent-child hierarchy.

(ii) What is a SiteMap?

  • A SiteMap is an XML file that defines the hierarchical structure of a website for navigation purposes.
  • It allows controls like Menu, TreeView, and SiteMapPath to automatically generate navigation links.
Purpose
  • Provides a centralized location for all navigation links.
  • Helps maintain consistency across the site.
  • Works with SiteMapDataSource for automatic binding.
Structure Example

<siteMap>

  <siteMapNode title="Home" url="Default.aspx">

    <siteMapNode title="About Us" url="About.aspx" />

    <siteMapNode title="Contact" url="Contact.aspx" />

  </siteMapNode>

</siteMap>

How It Works
  • The root <siteMapNode> represents the home page.
  • Nested nodes represent child pages.
  • The SiteMapDataSource reads this file, and navigation controls use it automatically.
Example with TreeView

<asp:SiteMapDataSource ID="SiteMapDataSource1" runat="server" />

<asp:TreeView ID="TreeView1" runat="server" DataSourceID="SiteMapDataSource1"></asp:TreeView>

Advantages
  • Simplifies website navigation management.
  • Auto-updates menus when sitemap changes.
  • No need for hardcoding navigation links.

Question 3

(i) List any three Validation Controls. Explain all with an example.

ASP.NET provides Validation Controls to ensure that user input in web forms is valid before processing. They reduce errors and improve data integrity.

1. RequiredFieldValidator
  • Ensures that the user does not leave a field blank.
  • Used for mandatory fields like username or email.

Example:

<asp:TextBox ID="txtName" runat="server"></asp:TextBox>

<asp:RequiredFieldValidator ID="rfvName" runat="server"

    ControlToValidate="txtName"

    ErrorMessage="Name is required."

    ForeColor="Red">

</asp:RequiredFieldValidator>

Explanation: Displays an error if txtName is empty.

2. RangeValidator
  • Ensures that the input falls within a specified range (numbers, dates, strings).

Example:

<asp:TextBox ID="txtAge" runat="server"></asp:TextBox>

<asp:RangeValidator ID="rvAge" runat="server"

    ControlToValidate="txtAge"

    MinimumValue="18" MaximumValue="60"

    Type="Integer" ErrorMessage="Age must be between 18 and 60."

    ForeColor="Red">

</asp:RangeValidator>

Explanation: Valid only if age entered is between 18 and 60.

3. CompareValidator
  • Compares the value of one input control with another or a fixed value.

Example:

<asp:TextBox ID="txtPassword" runat="server"></asp:TextBox>

<asp:TextBox ID="txtConfirm" runat="server"></asp:TextBox>

<asp:CompareValidator ID="cvPassword" runat="server"

    ControlToValidate="txtConfirm"

    ControlToCompare="txtPassword"

    ErrorMessage="Passwords do not match."

    ForeColor="Red">

</asp:CompareValidator>

Explanation: Displays an error if passwords don’t match.

Other Common Validators (for reference)
  • RegularExpressionValidator: Ensures input matches a specific pattern (e.g., email format).
  • CustomValidator: Defines custom validation logic via code-behind.
  • ValidationSummary: Displays all validation messages together.

(ii) Differentiate between GridView and DataList Controls

Feature

GridView Control

DataList Control

Layout

Fixed table layout (rows and columns).

Fully template-based layout (custom HTML).

Editing

Built-in editing, deleting, paging, and sorting.

No built-in editing; requires manual code.

Binding

Uses DataSource (like SqlDataSource).

Uses templates for data binding.

Performance

Heavier because of more features.

Lightweight and faster for large data.

Design Use

Tabular data (e.g., student records, products).

Flexible designs (e.g., photo gallery, card view).

Customization

Limited to styles and templates.

Highly customizable layout via templates.

Example (GridView):

<asp:GridView ID="GridView1" runat="server" AutoGenerateColumns="true"></asp:GridView>

Example (DataList):

<asp:DataList ID="DataList1" runat="server">

    <ItemTemplate>

        <%# Eval("Name") %> - <%# Eval("Price") %>

    </ItemTemplate>

</asp:DataList>

Summary:

  • GridView is ideal for structured data and admin panels.
  • DataList is better for flexible, design-oriented layouts.

Question 4

(i) A long list of articles needs navigation with next/previous buttons. Which control will you apply and why?

Control Used: DataPager (used with ListView control).

Explanation:

When displaying a large list of records (e.g., articles, posts), loading all items at once causes slow performance.
The DataPager divides the data into smaller pages and provides navigation using Next, Previous, or Page numbers.

Example:

<asp:ListView ID="ListView1" runat="server" DataSourceID="SqlDataSource1">

    <ItemTemplate>

        <h3><%# Eval("Title") %></h3>

        <p><%# Eval("Content") %></p>

    </ItemTemplate>

</asp:ListView>

<asp:DataPager ID="DataPager1" runat="server" PageSize="5" PagedControlID="ListView1">

    <Fields>

        <asp:NextPreviousPagerField ShowNextPageButton="true" ShowPreviousPageButton="true" />

    </Fields>

</asp:DataPager>

Why DataPager
  • Allows efficient navigation for long datasets.
  • Reduces page load time.
  • Supports both numeric and button-based paging.

(ii) Explain how to filter data using QueryExtender

QueryExtender is used with LinqDataSource to apply filtering, searching, or sorting operations without writing custom LINQ code in C#.

Purpose
  • Provides a declarative way to filter or search data.
  • Works at the data source level, improving efficiency.
Example:

<asp:LinqDataSource ID="LinqDataSource1" runat="server"

    ContextTypeName="SampleDataContext"

    TableName="Products">

</asp:LinqDataSource>

<asp:QueryExtender TargetControlID="LinqDataSource1" runat="server">

    <asp:SearchExpression DataFields="ProductName" SearchType="Contains" SearchTerm="Phone" />

</asp:QueryExtender>

<asp:GridView ID="GridView1" runat="server" DataSourceID="LinqDataSource1"></asp:GridView>

Explanation:
  • LinqDataSource fetches data from the Products table.
  • QueryExtender filters rows where ProductName contains the term “Phone”.
  • The GridView displays only filtered data.
Advantages
  • No need to write code-behind queries.
  • Improves performance through efficient LINQ filtering.
  • Useful for search bars, filters, and category-based views.
